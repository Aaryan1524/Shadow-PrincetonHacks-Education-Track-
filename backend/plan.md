 Context

 Building the complete FastAPI backend for Shadow, an AR coaching app. Dual-AI architecture:
 - Claude claude-sonnet-4-6 → visual "mind" (generate-steps, verify-step) — best vision model
 - Gemini 2.0 Flash → low-latency conversational coach (coach endpoint) — fast dialogue

 Backend is a clean slate. iOS Swift app exists but has no network layer yet.

 ---
 6 API Endpoints

 Registration order matters — static before parameterized.

 1. POST /lessons

 - JSON body: LessonCreateRequest
 - Generate uuid4 for lesson.id + each step.id, ISO8601 created_at
 - Save to disk, return full Lesson

 2. GET /lessons

 - Returns list_all_lessons() (empty list if none, never 404)

 3. POST /lessons/generate-steps ← MUST register before GET /lessons/{lesson_id}

 - multipart/form-data: frames: List[UploadFile], task_description: str = Form(...)
 - Uses Claude claude-sonnet-4-6, max_tokens=1000
 - base64-encode all frames → image content blocks
 - Ask Claude: respond ONLY with JSON array [{instruction, success_criteria}]
 - Parse JSON (regex fallback for preamble), assign uuid4 + order to each Step
 - Return: {"suggested_steps": [Step]}

 4. GET /lessons/{lesson_id}

 - load_lesson(lesson_id) → Lesson or 404

 5. POST /sessions/{lesson_id}/verify-step ← HOT PATH (called every 2s)

 - multipart/form-data: frame: UploadFile, step_index: int = Form(...)
 - Validate step_index → 400 if out of range
 - Uses Claude claude-sonnet-4-6, max_tokens=300 (low latency)
 - Content: [reference_image if exists] + [live frame] + [prompt + JSON schema]
 - Parse JSON with regex fallback
 - On ANY failure → safe CoachResponse fallback (never crash, iOS is polling)

 6. POST /sessions/{lesson_id}/coach

 - JSON body: CoachRequest
 - Validate step_index → 400 if out of range
 - Delegates to Gemini 2.0 Flash via coach_conversation() in agents module
 - Return CoachConversationResponse

 Shadow — Voice Coaching WebSocket Plan                                                                                                                                                               
                                                                                                                                                                                                      
 Context                                                                                                                                                                                              
                                                                                                                                                                                                      
 Adding real-time two-way voice coaching to the existing Shadow backend. When the learner taps "Start Learning", iOS opens a WebSocket to the backend, streams mic audio, and receives spoken         
 coaching back. Claude's /verify-step continues running every 2s in parallel — its result is sent to the voice coach for context so Gemini knows what the learner is doing visually.                  

 All 6 existing HTTP endpoints remain completely untouched.

 ---
 Architecture

 iOS (AVAudioEngine)
   │  PCM audio chunks → {"type":"audio_chunk", "data":"<base64>"}
   │  Silence detected → {"type":"end_of_speech"}
   │  Every verify-step result → {"type":"context_update", ...}
   ▼
 ws://backend/ws/sessions/{lesson_id}
   │  Accumulate PCM buffer
   │  On end_of_speech: build WAV → Gemini 1.5 Flash (audio + context) → gTTS
   ▼
 {"type":"audio_response", "data":"<base64 MP3>", "reply":"text"}
   │
   ▼
 iOS (AVAudioPlayer) → speaks coaching to learner

 ---
 Step 1: Create backend/agents/voice_coach.py

 Two public functions:

 build_wav_from_pcm(pcm_bytes, sample_rate=16000) → bytes

 Wraps raw PCM-16-bit-mono bytes into a valid WAV file using stdlib wave module. No deps.

 transcribe_and_coach(...) → dict

 Blocking function (called via asyncio.to_thread from WS handler):
 1. Builds Gemini prompt with: lesson context + step instruction + success criteria + latest CoachResponse from /verify-step
 2. Sends WAV bytes + context text to gemini-1.5-flash as a single Content with two parts (audio blob + text)
 3. Gets coaching reply text back
 4. Runs gTTS on reply text → MP3 in BytesIO
 5. Returns {"reply": str, "audio_b64": str|None, "updated_history": list}

 Critical: conversation history uses "model" role (not "assistant") — Gemini rejects "assistant"

 gTTS is lazy-imported inside the function so a missing install fails gracefully (endpoint still returns text reply with "data": null)

 Message handling:

 │      Received       │                                                        Action                                                        │
 ├─────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

 ├─────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
 │ WebSocketDisconnect │ exit loop cleanly                                                                                                    │
 └─────────────────────┴──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

 On Gemini error: send {"type":"error", "message":"..."} — do NOT close socket, allow retry next utterance.

 ---
 WebSocket Protocol

 Client → Server:
 {"type": "audio_chunk", "data": "<base64 PCM>"}
 {"type": "end_of_speech"}
 {"type": "context_update", "step_index": 0, "coach_response": {CoachResponse fields}}
 {"type": "ping"}

 Server → Client:
 {"type": "ready"}
 {"type": "processing"}
 {"type": "audio_response", "data": "<base64 MP3 or null>", "reply": "text"}
 {"type": "error", "message": "text"}
 {"type": "pong"}

 Audio format:
 - Input: PCM 16-bit, 16kHz, mono (iOS AVAudioEngine default)
 - Output: MP3 (gTTS, base64 encoded)
